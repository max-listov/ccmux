## Зачем (мотивация владельца)
Несколько агентов работают над общей целью — нужна **координация между ними без участия человека
как передаточного звена**. Кейсы:
- **Разделение ответственности**: агент A доделал свой кусок → сам сообщает агенту B «готово,
  подхватывай», не дожидаясь, пока владелец руками перекинет.
- **Командная работа**: разные сессии (разные проекты/роли/машины) кооперируются по одной задаче.
- Типовой пример: агент-A (working) пилит своё; агент-B (idle) свободен. Когда A закончит — B должен
  получить пинг. **Но не пока A занят собой**, и **A при отправке не должен думать о состоянии B** —
  просто пишет, транспорт разбирается.

### ⭐ ГЛАВНОЕ для владельца (приоритет №1, важнее всего остального)
**Сообщение НИКОГДА не должно улететь агенту и перебить его во время текущей работы.**
Это ключевой инвариант всей фичи. Всё остальное (лента, рендер, курсоры) — вторично относительно
этого. Доставка работающему = порча его контекста = провал фичи.

## Решения владельца (зафиксировано 2026-07-19)
- **Охват**: флот-wide, **any-to-any**, в перспективе **кросс-машинно** (host-A ↔ host-B ↔ host-C).
  Смотрим глобально с самого начала, не «только пара сессий на одной машине».
- **Per-session вкл/выкл**: у каждой ccmux-сессии индивидуально включается/выключается доступ в чат.
  (Настраиваемо — не все сессии участвуют.)
- **Полный фарш сразу, «чисто красиво пиздецки»**: доставляемость + прочтение + входящие для
  конкретного агента (от кого) + **чёткий лог from→to** + рендер ленты.
- **Само-дисциплина — ДА** (агент не засыпает с непрочитанным, реагирует на своей границе).
- **Адресация по имени сессии** — 100% (`ccmux msg <target> "..."`). Ролей в ядре НЕ вводим.
- **Тег отправителя — 100%**: доставка = впрыск user-turn, неотличимый от ввода владельца →
  сообщение обязано быть помечено `[from <session>]`, чтобы получатель понимал: это межагентское, не человек.
- **Полный payload разрешён** (не ограничиваемся «звонок ≤500 симв»).
- Тест: создадим **2 тестовые сессии**, гоняем чат между ними, наджерим руками.

## Что уже есть в ccmux (транспорт готов на 80%)
- `ccmux send <name> "<text>"` → `sendKeysLiteral` + 150ms + Enter → **полноценный user-turn**,
  агент сразу реагирует. (src/commands/send.ts, src/tmux/tmux.ts)
- `ccmux restart <name> --then "<note>"` → детач-воркер + `waitReady` (модель поднялась И idle,
  2 замера) → доставка ноты. Надёжно кладёт сообщение после готовности пейна. (src/commands/lifecycle.ts)
- `ccmux list --json` → `.state` (idle/working/stopped/missing) из скрейпа пейна — **сигнал
  «занят/свободен»**, гейт «не перебивать».
- Демон и так тикает ~30с и скрейпит пейны на всём флоте — естественный дом для очереди/доставки.

Чего **нет** и надо построить: очередь «придержать пока working, отдать на грани working→idle»,
лента+курсоры, per-session флаг, тег отправителя, рендер, само-дисциплина.

## Референс-паттерн (сторонний проект, single-machine) — что подсмотрели
Три слоя, все РАБОТАЮТ, но с оговорками:
1. **Лента-CLI** (`chat.ts`): append-only JSONL + курсоры прочитанного per-reader. `send from to msg`,
   `read` (двигает курсор), `peek` (без сдвига), `wait --timeout` (блокирующий вотчер). Философия:
   чат = «звонок», детали — в таске бэклога; лимит 500 симв. → *владелец: лимит НЕ берём, payload полный.*
2. **Само-дисциплина (Stop-хук)**: не даёт сессии закончить ход при непрочитанном ИЛИ без взведённого
   вотчера (`exit 2` + stderr-инструкция → харнес не останавливает, отдаёт инструкцию агенту).
3. **Доставка (внешний per-role демон, pm2)**: `tail -F ленты` (мгновенно) + поллинг-бэкстоп; читает
   `ccmux list` state; **idle → `ccmux send`, stopped → `ccmux restart --then`, working → НЕ трогает**
   (поллинг добьёт на следующем idle). ← **ccmux уже транспорт в их дизайне.**

**Ключевой инсайт**: их Stop-хук нужен как костыль именно потому, что лента не знает состояния сессии.
У ccmux состояние (idle/working) — **первично**. Значит ccmux может ловить грань working→idle и
доставлять надёжно из демона — а Stop-хук становится не костылём, а осознанным вторым рубежом.

## Модель данных (черновик — уточнить после сабагентов)
- **Сообщение**: `{ id, ts, from, to, body, delivered: bool, deliveredAt, readAt }`. `from`/`to` = имена
  сессий. Тег `[from <from>]` добавляется при доставке в пейн.
- **Лента**: append-only JSONL. Вопрос: одна на машину? синхронизируется по флоту? (см. открытые вопросы).
- **Курсоры**: per-session «прочитано до». Локально на машине получателя.
- **Per-session chat-флаг**: где живёт — в реестре `~/.ccmux-sessions` (новое поле) или machine.json?
- **Единственный писатель** = демон (как у реестра сейчас) — уважать инвариант, atomic-write.

## Варианты реализации (развилки — тут неоднозначно, поэтому в инбокс)

### A. Тонкий примитив «доставка-на-idle», без ленты
`ccmux msg <target> "<text>"` — fire-and-forget; демон придерживает и отдаёт на idle-грани. Без
курсоров/лога. Плюс: минимализм, кросс-машинно почти даром. Минус: нет аудита/реплея, нет
«входящих». → **Владелец это отверг** — хочет полный фарш. Оставлено как «нижняя граница».

### B. Полный чат (ВЫБОР ВЛАДЕЛЦА): лента + курсоры + рендер + доставка-на-idle + тег + само-дисциплина
`ccmux msg <target> "<text>"` (отправка), `ccmux inbox <session>` (входящие/непрочитанное),
`ccmux chat log [--between A B]` (рендер ленты from→to), `ccmux chat on/off <session>` (per-session флаг).
Демон: очередь → доставка на working→idle → тег `[from …]` → отметка delivered/read.
Само-дисциплина: Stop-хук-подобный рубеж (агент не спит с непрочитанным).

### C. Фат: роли/каналы/broadcast/ACK
Роли, каналы, `to: all`, подтверждения. → **НЕ сейчас** (§31 — не плодить классификации; роли —
семантика приложения, не ccmux). Зафиксировать как возможное будущее, в ядро не тащить.

## Механизм «не перебивать» — как именно (главный узел, для сабагентов)
Кандидат-дизайн:
1. Отправка `ccmux msg` → запись в ленту со `delivered:false`. Отправитель свободен, состояние
   получателя не знает.
2. Демон на машине получателя (он владеет её tmux) в своём тике/вотч-цикле смотрит непрочитанные+недоставленные.
3. Гейт по state получателя:
   - `working` → **НЕ доставляем**, ждём.
   - `idle` → доставляем `ccmux send` (тег `[from …]`), ставим delivered.
   - `stopped` → `ccmux restart --then` (доставка после готовности).
4. **Анти-рейс working→idle** (критично): перед самой доставкой — свежая перепроверка state
   (не полагаться на 30с-снимок), возможно требование N idle-кадров подряд + короткий лок, чтобы
   между «idle» и «send» агент не начал новую работу (напр. фоновая задача до-ехала и разбудила его).
5. Латентность: 30с тик — многовато для «чата». Возможно нужен **отдельный быстрый watch-цикл**
   доставки (не путать с healing-тиком). Открытый вопрос.

## Открытые вопросы (нужно решить до кода)
1. **Латентность доставки**: хватает healing-тика 30с, или отдельный быстрый watch-loop? Насколько
   «мгновенным» должен быть чат?
2. **Кросс-машинная маршрутизация**: реестр видит весь флот, но демон рулит только СВОЕЙ машиной.
   Как host-A→host-B реально доставить? Есть ли уже канал между демонами, или нужен новый (и минимальный
   правильный: доставку делает демон машины-получателя, читая общую/реплицированную ленту)?
3. **Где живёт лента**: одна на машину (тогда кросс-машинно нужна репликация/пуш) или общий флотовый
   стор? Синк по образцу live-client (watch jsonl)?
4. **Определение состояний**: скрейп различает idle-после-работы vs idle-в-ожидании-ввода-человека vs
   краткий idle между тул-коллами vs пермишен-промпт vs plan-mode? Какие из них доставку ПОРТЯТ?
5. **idle + человек печатает**: получатель idle, потому что владелец сейчас будет печатать в него.
   `ccmux send` вклинится. Можно ли вообще детектить человека-в-процессе-ввода? Митигейшн?
6. **Multi-line payload**: `send` = литерал + 150ms + Enter. Встроенный `\n` не сабмитит раньше времени?
   Как фреймить многострочное/длинное?
7. **Порядок и батч**: 3 сообщения в очереди — доставить все на idle, по порядку? Доставка №1 флипает
   агента в working → №2/№3 корректно ждут снова?
8. **Само-дисциплина vs очередь**: Stop-хук на непрочитанном не конфликтует с «доставкой-на-idle»
   (двойная доставка)? Кто главный рубеж?
9. **Тестовый сетап**: 2 тестовые ccmux-сессии, сценарии — (a) получатель idle, (b) получатель working
   → ждёт idle, (c) получатель stopped → restart --then, (d) батч, (e) человек печатает в получателя.

## Выводы валидации (3 сабагента, 2026-07-19)

### 🔴 ГЛАВНЫЙ ВЫВОД (бьёт в приоритет №1): «НЕ перебивать» НЕ достигается скрейпом пейна
Всё состояние сейчас — один регекс по низу пейна:
`WORKING_RE = /[✱-✿] слово…|esc to interrupt/` → `working`, иначе `idle` (`src/agent/claude/pane.ts`).
Это **бинарно**, и «idle» молча включает состояния, куда впрыск user-turn = **порча**:
- **(d) пермишен-промпт** («allow this Bash? 1.Yes 2.No») — спиннер погас → скрейп видит `idle` →
  доставленный текст **отвечает на меню/аппрувит операцию, которую агент не выбирал**. ⚠️ Под
  `permissionMode: auto` на серверах промпты всё равно всплывают на не-safe операции.
- **(e) plan-mode** («proceed? 1.Yes/2.No») — то же: авто-ответ на гейт плана, запускает выполнение.
- **(c) краткий кадр между тул-коллами / thinking↔acting** — спиннера в кадре нет → ложный `idle` →
  впрыск посреди хода.
Это **не гонка, а категориальная дыра модели**: перепроверкой N раз пермишен-промпт, выглядящий idle,
не лечится. **Вывод: гарантия «NEVER interrupt» на скрейпе — ложна по построению.**

**Что нужно вместо скрейпа (сходятся все три):** авторитетное состояние из **хуков Claude Code**,
устанавливаемых в самих чат-сессиях:
- `Notification`-хук → флаг `blocked-at-prompt` (пермишен/plan — позитивно ИСКЛЮЧАЮТСЯ из «safe idle»);
- `Stop` / `UserPromptSubmit` / `PreToolUse` → busy/idle.
Скрейп — только вспомогательный сигнал, не авторитет.

### Осуществимость (сабагент 1) — что подтвердилось на коде
- **Демон не сканирует state вообще** (`ensureOnce` только хилит). «Доставка на idle-грань» = НОВЫЙ код;
  30с-тик как часы чата не годится → **отдельный быстрый loop / событийная доставка на переходе state**,
  держащий `Map<name,lastState>`, сканирует только chat-сессии с непрочитанным (форк `capture-pane`
  только когда что-то в очереди — в покое почти бесплатно).
- **Транспорт send-keys `-l` небезопасен** для multi-line/длинных (шелл-мета — ОК, там argv без шелла;
  ломается именно `\n` = ранний сабмит + длина). Правильно: `load-buffer` (stdin) → `paste-buffer -p`
  (bracketed paste, многострочное одним блоком, не сабмитит) → один `Enter`. Добавить `pasteBuffer`
  рядом с `sendKeysLiteral`.
- **Кросс-машинного канала НЕТ** (реестр строго локальный). Мин-правильно: SSH-push `ccmux _enqueue`
  в спул демона машины-получателя; нужна **fleet-directory** (имя→машина; имена уникальны лишь внутри
  машины → адресация `<rcPrefix>-<name>` + мапа prefix→host). Синк-лента (chezmoi-стиль) — ОТВЕРГНУТА
  (last-write-wins молча теряет параллельные сообщения).
- **Хранилище:** chat-флаг → поле `Session` (прецедент `permissionMode`, `sessions.ts:58`); лента →
  **отдельный** append-файл (реестр пишется full-rewrite → затрёт + гонки с re-pin форка); курсоры →
  крошечный daemon-owned файл. Отправитель только append-ит per-message спул-файл, **демон —
  единственный сериализатор** в ленту.
- **Бонус:** `CCMUX_SESSION` уже в env каждой сессии (`launch.ts:76`) → тег `[from …]` бесплатно.

### Атака на инвариант (сабагент 2) — остаточные риски даже в лучшей версии
Закрываемо: TOCTOU-окно (пере-скан прямо перед вставкой + сверка mtime транскрипта `lastActivityMs`
не двигался), multi-line (bracketed paste), очередь (on-disk FIFO, **edge-triggered с гистерезисом**:
одно сообщение на цикл idle→working→idle, транзакционный pop). Дубль push/pull — выбрать ОДИН авторитет.
**Неустранимо (владелец должен принять явно):**
- **A** — ненулевое TOCTOU-окно между финальной проверкой idle и приходом Enter (хук/шедулер/фоновая
  задача могут начать ход) — только сужается, не обнуляется.
- **B** — `send-keys` в общий readline **не атомарен против печатающего человека**; детектить человека
  в процессе набора нельзя → защита только политикой: `tmux list-clients` — есть аттач → отказ/очередь
  (функциональная плата: пока владелец приаттачен, сообщения ждут).

### Prior art / фит (сабагент 3)
- **Нативно в Claude Code — НЕТ** (громкий незакрытый запрос: issues #24798/#37213/#33788/#28229).
- **Почти-точный аналог есть плагином** — `yilunzhang/claude-code-inter-session` (`send/broadcast/list`
  по имени, лог payload, WS-шина): **НО доставляет через Monitor-tool, который ПЕРЕБИВАЕТ** («acted on
  as instructions by default»), single-machine. Т.е. самый популярный солюшн делает ровно то, что твой
  приоритет №1 запрещает.
- Мульти-агентские фреймворки (AWS CAO, tmux-bridge-mcp, Coral) — все либо перебивают, либо оркестратор,
  никто не «persistent self-healing fleet».
- **Вывод: «queue-until-idle / доставка на idle-грань» — реально незанятая ниша, и это МОАТ ccmux:**
  корректная не-перебивающая доставка ТРЕБУЕТ авторитетного state сессии, которым владеет только ccmux
  (демон + `list` state + `send`/`restart --then` + heal-loop). Усиливает нишу + мост к live-client.
- **НО: риск нативного вытеснения** — сигнал «teammates/mailbox/next-turn injection» (cuinspace) намекает,
  что Anthropic может выкатить сам → **строить ТОНКО, дёшево бросить**. Владеть транспортом+доставкой+лентой;
  роли/треды/task-linking — НЕ в ядро (app-level; §31).
- **Новые обязательные гардрейлы v1:** анти-шторм / петля A→B→A (rate-limit + per-session budget +
  loop-detection) — пререквизит, не экстра. Фрейминг сообщения как **peer-сигнал, а не команда** (иначе
  агент бросит свою задачу). Trust: chat default-OFF + возможен allowlist; учесть `bypassPermissions` + root-guard.

### 🎯 Два главных вывода из референсов
1. **Ниша ccmux подтверждена ещё жёстче: НИ ОДИН из трёх не решает «не перебивать работающего».**
   Греп по всем их README на `interrupt|mid-task|turn boundary|when idle` — **пусто**. Они защищаются
   *семантикой* (reaction-policy) или *обвязкой* (ACL/persist), а не **таймингом доставки**. Твой
   приоритет №1 — реально незанятое место. ccmux уникален тем, что владеет pane-state.
2. **⭐ Открылся транспорт ЧИЩЕ, чем скрейп+send-keys (который сабагент 2 признал нерешаемым).**
   Два из трёх доставляют не через впрыск в пейн, а через **CC-нативный async-канал**:
   - **Monitor-tool** (inter-session): долгоживущий субпроцесс, его stdout CC показывает агенту;
   - **MCP server-notification** (relay): `server.notification({method:"notifications/claude/channel"})`.
   Это убирает и скрейп пейна, и readline-гонку, и multi-line-проблему — **структурированная доставка**.
   **НО** оба выглядят как *wake/interrupt* (ms-латентность, «wakes the session»), а не «придержать до
   границы хода». → **Решающий незакрытый вопрос:** в какой момент CC показывает модели channel-notification
   / Monitor-строку — **посреди хода (перебивает) или на границе следующего хода (безопасно)?** Если на
   границе — это и есть чистое «не перебивать» БЕЗ скрейпа, и ccmux мог бы **быть/встроить MCP-relay**,
   а не печатать в пейн. Проверяется эмпирически на 2 тестовых сессиях.

### Пересмотр транспортных вариантов (3 парадигмы, не 1)
- **T1 — впрыск в пейн (`send-keys`)**: текущий примитив ccmux + tmux-bridge. Просто, но перебивает +
  хрупко (сабагент 2). ccmux уникально может гейтить по своему pane-state — но гейт вероятностный.
- **T2 — CC Monitor-tool** (inter-session): stdout субпроцесса → агент. Структурно, но wake-based.
- **T3 — MCP server-notification** (relay): структурно, персист, кросс-машина, ACL. Тайминг = внутри CC.
- **Гипотеза дизайна:** если T3/T2 отдаются модели на границе хода — берём его как основу (снимает весь
  узел «не перебивать»), а send-keys/`restart --then` оставляем лишь для stopped-сессий. Это надо
  **проткнуть реальностью** до выбора архитектуры (правило «дешёвый пробник до теории»).

## 🏗️ План: миграция проверенного внешнего референса → ccmux-native (без потерь, только апгрейд)

### Главный архитектурный принцип (взят у референса, обобщён)
**Лента = единственный источник правды. Доставка = fan-out по СТОКАМ, у каждого свой персист-курсор.**
Референс это уже так делает: CLI пишет `CHAT.jsonl` → мост вотчит файл → фанит каждую новую строку в
стоки (WS + Telegram), курсор доставки переживает рестарт (досылает backlog за простой, дедуп, единый
писатель). У ccmux это становится ядром: **любой канал вывода — просто сток, подписанный на ленту.**

Стоки:
1. **pane** (agent↔agent) — ядро: доставка в пейн получателя, **menu-safe гейт** (хук `Notification`).
2. **telegram** — зеркало переписки агентов в бота юзера (группа/личка/топик).
3. **ws live-client** — будущее, сходится с `2026-06-09-ccmux-live-web-mobile-client.md` (тот же watch-ленты).

Стоки независимы и со своими курсорами: TG-мост лежал → поднялся → досылает пропущенное; pane-доставка
своим курсором. Ровно паттерн `ChatBridge` референса (persist-курсор `delivered`, resync при усадке ленты).

### Где логируется (полное сохранение для дебага/вывода/экспорта)
- **`~/.ccmux-chat.jsonl`** — append-only ЛЕНТА, **каждое** сообщение навсегда: `{id, ts, from, to,
  body, task?, delivered, deliveredAt, readAt}`. Это и есть полный лог — ничего не трим (или явный
  retention-конфиг). Дебаг/вывод/экспорт = чтение/копия этого файла.
- **`~/.ccmux-chat-cursors.json`** — курсоры: per-sink `delivered` (pane/tg/ws) + per-session `read`.
  Демон — **единственный писатель** (как `ChatBridge`). Крошечный, atomic-write.
- Рендер: `ccmux chat log [--between A B] [--since T]`, `ccmux inbox <session>` (непрочитанное).
- Лента отдельно от реестра `~/.ccmux-sessions` (реестр — full-rewrite, затрёт; см. вывод сабагента 1).

### Telegram-мост — «любой юзер ccmux вводит токен → переписка его агентов летит к нему»
Config **per-machine** (`machine.json`, как весь конфиг ccmux):
```
telegram: {
  botToken: "<от @BotFather>",     // секрет юзера
  chatId: "<numeric group/DM id>", // куда слать
  topicId?: <message_thread_id>,   // опц. — конкретный ТОПИК супергруппы
  routing?: [{ session|pair, chatId, topicId }]  // опц. — разные сессии/пары в разные топики/чаты
}
```
- **chat→TG (сток):** на каждое новое сообщение ленты — в бота `[<from> → <to>] <текст>`, в `topicId`
  если задан. (Референс шлёт ещё баннер-картинку направления + caption; для ccmux v1 — текст, баннеры
  опц. позже.) **Fail-soft:** нет токена/чата → просто не зеркалим, ядро работает (как у референса).
- **TG→chat (обратка, bidirectional — как у референса):** юзер отвечает боту в той же
  группе/топике → ingest пишет это в ленту как сообщение от `owner`/`human` → доставляется целевому
  агенту тем же pane-стоком. **Так юзер рулит агентами с телефона.** (Демон держит long-poll/webhook бота.)
- **Гибкость маршрутизации** (то, что просил): либо всё в одну группу, либо в личку, либо в конкретный
  топик конкретной группы; `routing` позволяет развести разные сессии/пары по разным топикам.

### Что сохраняем 1:1 (без потери) vs что улучшаем
| Из референса — берём как есть | ccmux-native улучшение |
|---|---|
| Append-only лента = источник правды | + это же полный лог для дебага/экспорта |
| Мост файл→стоки с **persist-курсором** (переживает рестарт, досылает backlog, дедуп) | + курсор per-sink, единый писатель = демон |
| Telegram-сток: чат+топик, fail-soft | + **per-user токен** (не один хардкод-админ) + routing по сессиям |
| TG→chat обратка (юзер отвечает из телеги) | + адресация ответа по имени сессии |
| «чат = звонок, детали в таске» + лимит длины | сохраняем дисциплину |
| Гейт доставки по состоянию сессии | + **menu-safe** (хук `Notification`) — закрыта дыра Пробника B |
| — (роли хардкод) | **имя сессии**, динамически |
| — (per-project, single-machine, 3 механизма, pm2-демон на роль) | **ядро ccmux, флот, один демон-авторитет, 0 лишних процессов** |
| — (всегда для зашитых ролей) | **per-session вкл/выкл** |

### Фазы миграции
- **Фаза 0** — модель данных + развилки (ниже) зафиксировать → «го».
- **Фаза 1 (ядро agent↔agent)** — лента + `ccmux msg`/`inbox`/`chat log`/`chat on|off` + pane-сток с
  **menu-safe гейтом** (хук `Notification` пишет флаг «на промпте») + тег `[from …]` + per-session флаг.
  Тест на 2 сессиях (сценарии из пробника, вкл. пермишен-меню).
- **Фаза 2 (Telegram-зеркало)** — telegram-сток (chat→TG) + config + fail-soft. Тут же — полный лог готов.
- **Фаза 3 (Telegram обратка)** — TG→chat ingestion (long-poll/webhook), ответ адресуется сессии.
- **Фаза 4 (флот + live-client)** — кросс-машинная маршрутизация (SSH-enqueue в демон-получатель) +
  WS-сток (сливается с live-client-таской).

## Развилки после валидации (для решения владельцем)
1. **Модель state доставки:** (принято де-факто как единственно корректная) — авторитет из **хуков CC**
   (Notification→blocked, Stop/UserPromptSubmit→busy/idle), скрейп вспомогательно. Ставит хуки в чат-сессии.
   Альтернатива — только скрейп — валидацией **отвергнута** (ложные idle на промптах). Подтвердить.
2. **Локально сначала или флот сразу?** Осуществимость+prior-art советуют **local-only v1** (паритет с
   inter-session, кросс-машина = отдельный net-new SSH+fleet-directory слой) → кросс-машина фаза 2.
   Владелец хотел «глобально сразу» — развилка сложности, решить явно.
3. **Латентность:** событийная доставка на переходе state (не 30с-полл).
4. **Push xor pull:** демон-доставка на idle-грань (push) ИЛИ Stop-хук-подтяг (pull) — ОДИН авторитет,
   транзакционный pop. Само-дисциплину-хук сделать вторым рубежом ОПЦИОНАЛЬНО, не дублируя доставку.
5. **Принять остаточные A и B** (TOCTOU-окно; отказ доставки при аттаче человека) — да/нет.
6. **Гардрейлы v1** (rate/budget/loop, chat default-off, фрейминг peer-сигнала) — фиксируем как
   non-negotiable для релиза.

Далее: по этим развилкам — чистый дизайн одним заходом (модель данных + слои + гарантия), и по «го» — код.
