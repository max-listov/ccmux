## Пробел 1 — Живучесть await-лупа роутера (КРИТИЧНО)

### Проблема (корень)
Роутер просыпается **только на входящее сообщение**. Ответ таргета — ручной report-back. Если таргет
**доделал, но не отчитался** (или завис) — роутер **молча висит**, задача не доводится. Это прямая дыра
против постановки «роутер сам доводит, меня не беспокоит». Нужна **опора по времени**, а не только по
событию.

### Примитив: отложенная по времени доставка
- Поле **`notBefore: string | null`** (ISO-8601) на `ChatMessageSchema` (default null).
- **`ccmux msg --after <сек>`** → `notBefore = now + сек`.
- Демон доставляет сообщение только когда `now >= notBefore` (пока рано — пропускает). Латентность
  доставки после срока ограничена 3с-циклом `chatDeliveryLoop` — приемлемо для минутных watchdog'ов.

### Что это ЛОМАЕТ и как чинить (центральная развилка)
Watchdog = роутер шлёт **себе** `msg <self> --after N`. Это будущее-датированное сообщение ложится в
ledger адресованным роутеру. При текущей модели (Фаза 1/2 — defer «держится на монотонном курсоре»)
оно **заблокирует последующие немедленные сообщения роутеру** — в т.ч. report-back таргета застрянет
за собственным watchdog'ом роутера. Значит **notBefore НЕЛЬЗЯ держать на курсоре — он обязан быть
пропускаемым.** Это же — латентная дыра N1 (head-of-line), которую мы отложили в Фазе 1.

**→ Рефактор модели доставки (закрывает и N1):** `deliverPending` на два трека на получателя:
- **Immediate-трек** — монотонный `delivered`-курсор по сообщениям, которые НЕ `defer` и НЕ
  будущее-`notBefore`; по порядку (текущее поведение).
- **Conditional-трек** — `defer` и `notBefore`-сообщения, доставляются **по message-id** когда условие
  выполнено (`defer`→стабильный idle / Stop-hook; `notBefore`→срок прошёл), НЕЗАВИСИМО от позиции в
  ledger; дедуп — существующий ack-log (`store.ts`). Курсор-скан **пропускает** conditional-сообщения
  (продвигается мимо — их владелец conditional-трек).
- Итог: head-of-line убран и для defer, и для notBefore.

### Протокол watchdog'а роутера (в `promptModules.ts` routerModule)
На каждый диспатч роутер:
1. Взводит watchdog: `ccmux msg <self> --after N --task <routing-id>`.
2. На срабатывании читает `ccmux transcript <target> --json` и решает:
   - **доделал-но-молчит** (idle + работа видимо завершена) → закрыть, доложить owner;
   - **ещё работает** → перевзвести (`--after N` снова);
   - **завис / idle слишком долго без прогресса** → эскалация owner;
   - **уже закрыто** (routing-id разрешён) → **no-op** (идемпотентно).

### Хранение routing-id / трекинг / идемпотентность
- **routing-id едет в поле `task`** (переиспользуем — новое ccmux-поле не нужно).
- Трекинг открытых диспатчей (routing-id → target + done-критерий + счётчик re-arm) — **файл состояния
  в рабочей папке роутера** (роутер — агент, пишет файлы). **ccmux остаётся транспортом**, не знает про
  routing-id.
- **Durability:** будущее-watchdog персистит в append-only ledger; файл состояния — в папке роутера →
  оба переживают рестарт/daemon-heal (срок пришёл → демон доставит; роутер читает файл → продолжает).
- **Идемпотентность:** если таргет отчитался ДО watchdog'а — routing-id уже закрыт → срабатывание
  watchdog'а = no-op. Поздний/дублирующий будильник безвреден.

### Развилки
1. **Объём рефактора доставки:** полный two-track split (**рекомендую** — закрывает N1) vs. минимальный
   «только notBefore пропускается, defer остаётся на курсоре». Рекомендую полный — консистентно и лечит N1.
2. **`msg cancel`?** — отменять висящий watchdog при раннем отчёте vs. полагаться на идемпотентный no-op.
   **Рекомендую идемпотентность** (проще; лишний будильник дёшев). `cancel` — опционально позже, если
   холостые пробуждения станут заметны.
3. **Дефолт таймаута N:** фикс-дефолт vs. роутер выбирает под задачу. **Рекомендую роутер-выбор** (он
   знает размер задачи), с полом/потолком в протоколе + **cap на re-arm (напр. ≤3, потом эскалация)** —
   против бесконечного ожидания зависшего таргета.
4. **notBefore + defer на ОДНОМ сообщении?** watchdog = notBefore (не defer); диспатч = defer (не
   notBefore) — независимы. Conditional-трек тянет любой набор (оба условия должны выполниться). Задать явно.

### Объём / якоря
- `config/schema.ts:166` `ChatMessageSchema` — `+ notBefore`.
- `commands/msg.ts` — `--after <сек>` → notBefore.
- `chat/deliver.ts:54` `deliverPending` — two-track рефактор; conditional-доставка включает `now >= notBefore`.
- `chat/store.ts` — ack-log уже дедупит defer по id; распространить на notBefore (тот же id-keyed дедуп).
- `agent/promptModules.ts` `routerModule` — добавить пункты watchdog/re-arm/cap/idempotency.
- Конвенция файла состояния роутера — только в промпте/доке (пишет агент).

### Риски
- Рефактор трогает рабочий Фаза-1/2 defer-путь → **пере-прогнать полный e2e** (defer держится при работе,
  доставка на idle, hook на end-turn) — не должно регрессировать.
- Бесконечный re-arm при вечно-зависшем таргете → cap + эскалация (развилка 3).
- Часы: notBefore — wall-clock ISO, `now` из Date; отправитель и демон на одной машине (общие часы) — ок;
  cross-machine — позже.
- Рост ledger от watchdog-самосообщений усиливает R4 (полное чтение ledger на тик/turn) — см. «Мелочи».
