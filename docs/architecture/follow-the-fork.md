# Follow the fork

## Проблема (реальный инцидент 2026-07-14)

Инвариант «фикс-uuid на сессию» **неверен для Claude Code**: при переполнении контекста
(и, возможно, в других недокументированных случаях) отправка следующего сообщения
**форкает беседу** — новый session id, новый jsonl с скопированным хвостом старого
(включая событие `custom-title`), исполнение под демоном Claude (`bg-pty-host`,
v2.1.200+). Argv процесса в пейне при этом всё ещё говорит `--resume <старый-uuid>`.

Наблюдалось вживую на managed-сессии: беседа уехала `e8f056d3 → 711f8574`. Все
jsonl-производные данные ccmux (превью, активность, транскрипт, CTX-fallback) замёрзли
на старом файле, живая беседа всплыла дублем в external, а рестарт поднял бы беседу
трёхдневной давности. Официальные доки форк не документируют (и местами утверждают
обратное) — поэтому детект строится **без предположений о триггере** форка.

## Модель

Единственный источник правды о «текущей беседе сессии» — **сам транскрипт-каталог**:

- Идентичность — **RC-имя**: ccmux каждый запуск передаёт `-n <rcName>`
  (`<rcPrefix>-<name>`), Claude пишет его в jsonl событием `custom-title`, и **форк
  наследует его в первых строках** материализованного файла.
- «Текущая беседа сессии X» = jsonl в проектной папке X, чей head несёт title X и чьё
  **последнее сообщение** новее всех (timestamp записи, не mtime — mtime врёт: его
  бампает открытие в desktop-аппе).

## Реализация

| Слой | Что делает |
|---|---|
| `src/agent/claude/fork.ts` | `detectFork(s, m, rcTitle, takenUuids)` — скан проектной папки, гварды, выбор новейшего титулованного кандидата |
| `src/agent/index.ts` | опциональный хук контракта `AgentProvider.detectFork` + обёртка `forkedUuid(s, m, all)` (сама исключает uuid других managed-сессий) |
| `src/config/sessions.ts` | `updateSessionUuid` — атомарная перепривязка одной строки реестра |
| `src/commands/ensure.ts` | каждый heal-пасс (демон, 30с): сначала follow-fork, потом решение о старте — рестарт всегда резюмит живую беседу |
| `src/agent/claude/writers.ts` + `tui/discover.ts` | `externalResumingUuids` — external-дискавери игнорирует процессы ВНУТРИ managed-пейнов (предок — `ccmux _run`): после форка argv пейна всё ещё несёт старый uuid, и без фильтра мёртвая беседа всплывала бы «живым» external-дублем; форк-в-переходе (bg-pty-host — тоже потомок пейна) — забота follow-the-fork, не external |
| `src/util/readLines.ts` | общие байтовые читалки head/tail (вынесены из agent/index + tui/discover — нужны и форк-детекту, без циклов импортов) |

Демон — **единственный писатель** реестра при форке (TUI/list только читают и
подхватывают на следующем полле, ≤30с). Лог: `follow fork: conversation moved`.

## Гварды (кандидат обязан пройти ВСЕ; сомнение → не трогаем)

1. `<uuid>.jsonl` в проектной папке сессии; не pinned-uuid и не uuid другой managed-сессии.
2. **stat-префильтр**: mtime кандидата > времени последнего сообщения pinned-файла
   (mtime ≥ последней записи, значит старый по mtime файл новых сообщений не содержит) —
   30-секундный тик демона обходится stat'ами, не чтением всех транскриптов.
3. Head (16КБ) несёт `custom-title == rcName` сессии. Две сессии, живущие в одной
   папке, имеют разные RC-титулы — чужой форк не присвоится. Ручной
   резюм древней беседы с тем же `-n` дописывает title в ХВОСТ её файла — head-скан
   его не видит, false-positive нет.
4. Последнее сообщение кандидата **строго новее** pinned (равенство = не переезд;
   после перепривязки новее нет → стабильность, без флаппинга).
5. Из нескольких прошедших (форк форка) побеждает новейший.

## Что чинится автоматически

Перепривязка реестра лечит всё разом: превью/активность/сортировку карточки,
fullscreen-транскрипт, CTX-fallback, резюм при рестарте/ребуте; external-дубль
форка исчезает (uuid стал managed), а СТАРЫЙ uuid не всплывает external-дублем
благодаря pane-фильтру `externalResumingUuids`. Старые jsonl остаются на диске
нетронутыми.
