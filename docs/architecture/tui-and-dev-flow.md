# TUI (Ink) и dev-флоу

## Что это
`ccmux` без аргументов на TTY → интерактивный терминальный UI на **Ink** (React → Yoga →
ANSI; тот же класс, что Claude Code). Категория — **inline-native** по умолчанию (живёт в
потоке терминала, остаётся в истории), **fullscreen** по флагу (`-f`, alt-screen + листаемый
транскрипт справа). Клавиша `f` переключает на лету.

Структура — `src/tui/` (DRY: общие `format`/`components`/`hooks`, два `views/` отличаются
только layout-обёрткой). Подробности компонентов — в коде.

## Dev / Prod / self-update
Bun-ccmux **заменил** bash-ccmux локально (bash-демон убит, плист `.legacy`, бинарь → `ccmux-legacy`
как fallback). Два режима, **развязаны** — правка дев-исходника НЕ трогает прод-демон, который рулит
боевыми сессиями:

| Режим | Команда | Что запускает |
|---|---|---|
| **DEV** | `ccmux-dev` | `bun run <repo>/src/cli.ts` (исходник, `bun tui:hot` = `--hot` для HMR) — разработка |
| **PROD** | `ccmux` | `bun ~/.ccmux/app/ccmux.js` (бандл ~1.7МБ) — боевой; демон `com.ccmux.daemon` крутит ЕГО |

Лаунчеры: `~/.local/bin/ccmux-dev` → исходник; `~/.local/bin/ccmux` → бандл.

**Инвариант: ОДИН демон (прод).** Дев демон НЕ ставит → нет коллизии двух ensure-циклов (делить
реестр двум демонам нельзя — был реальный баг: bash+наш писали разный формат → мусорные сессии).
Оба режима читают один `~/.ccmux-sessions` (JSONL Session-schema) — чтение из обоих безопасно.

### Бандл + self-update
- **Сборка** — dev-скрипты в `package.json` (НЕ команды тулзы — `ccmux` их не содержит, в клиентский
  бандл они не попадают): `bun run stage` → `bun build --target=bun --external react-devtools-core` →
  `~/.ccmux/staged/ccmux.js` (для локального теста апдейта). Логика — `scripts/release.ts`.
  (`react-devtools-core` external: Ink тянет его только в СВОЁМ dev-режиме, в проде не нужен.)
- **Апдейт**: `ccmux update` — **локальный staged ПЕРВЫМ** (тестим сборку локально), иначе remote (`releaseUrl`+sha256).
  Atomic swap `~/.ccmux/app/ccmux.js` (+`.bak`) → bounce демона (`kickstart -k`). **Сессии переживают** (tmux
  независим от демона); каждый `_run` подхватывает новый код на следующем рестарте. Откат: `ccmux update --rollback`.
- Версия — в шапке TUI (`ccmux vX.Y.Z · fleet`). Пути: `src/config/paths.ts`. SELF_ARGV (`src/env.ts`)
  различает compiled-бинарь / dev-исходник / bundle → плист печёт `bun ~/.ccmux/app/ccmux.js daemon`.

### Раздача на флот (GitHub Releases, без своего бэкенда)
- **Публикация** — dev-скрипт `bun run release:publish [notes]` (НЕ команда тулзы; `scripts/release.ts`,
  только из репо — у клиента нет ни репо, ни package.json) — build → sha256 → манифест
  `release.json` (`{version,notes,sha256,url}`) → `gh release create vX.Y.Z --draft` с тремя ассетами
  (`ccmux.js`, `release.json`, `install.sh`) → `--draft=false`. Draft→upload→publish = релиз появляется
  **атомарно** (полу-залитого флот не видит). Манифест `url` указывает на **версионный** ассет
  (`releases/download/vX.Y.Z/ccmux.js`) → пара манифест+бандл неразрывна, latest-гонок нет. Версия
  иммутабельна — повторный `--publish` той же версии отказывает (бамп `package.json`).
- **Подписка флота**: машины ставят `releaseUrl = .../releases/latest/download/release.json` +
  `autoUpdate:true`. Демон каждые `updateCheckInterval` (деф. 300с; серверам ставим 60с) сравнивает
  semver, тянет, **sha256-verify → preflight → swap**. Анонимный HTTPS + CDN GitHub, токены не нужны.
- **Защита от кривого релиза, 2 слоя** (`update.ts` + `util/bootGuard.ts`): (1) **preflight** —
  `bun <candidate> version` обязан вернуть верную версию ДО свапа (ловит не-загружающийся бандл,
  live не трогается); (2) **boot-guard** — счётчик стартов демона (`~/.ccmux/boot-attempts`):
  3 старта без успешного ensure-тика → возврат `.bak` + exit≠0 → boot-юнит поднимает старый бандл.
  Демон чистит счётчик после первого хорошего тика. Откат вручную: `ccmux update --rollback`.
- **Новый клиент** — одной командой: `curl -fsSL <repo>/releases/latest/download/install.sh | bash`
  (`scripts/install.sh`): ставит bun при отсутствии, качает бандл + sha256-verify, шим в `~/.local/bin/ccmux`,
  `ccmux install --rc-prefix … --release-url <manifest>` (пишет releaseUrl + autoUpdate). Override
  источника: `CCMUX_REPO=owner/name`.

### Системный лог (`~/.ccmux/ccmux.log`)
JSON-строки, уровни `debug<info<warn<error`, порог `logLevel` в machine.json (деф. `info`, демон
перечитывает живьём — флипнуть в `debug` без рестарта). Ротация по размеру (5MB × 3 ≈ 15MB cap,
фикс — не конфиг, чтобы лог не съел диск сервера). События: daemon up/stop, heal (сессия упала→рестарт),
session new/start/stop/rm, update seen/applied/failed/rolled-back, boot-guard revert. `src/util/log.ts`.

## Реальные данные
`ccmux` (бандл) и `ccmux-dev` (исходник) читают живой `~/.ccmux-sessions` — это не мок.
`rcPrefix` дефолтится в `local` (свежая машина без `machine.json` просто работает; `install`
пишет реальный local|dev|prod). Изоляция dev-реестра при нужде — `CCMUX_SESSIONS=…`.

## Правки → проверка
Hot-reload:
- `bun tui:hot` = `bun --hot src/cli.ts` — Bun перезагружает модули на изменение (для TUI
  экспериментально: raw-mode может мигнуть; если глючит — обычный рестарт).
- Практичный цикл: правка → `q` → перезапуск (мгновенный). Для fullscreen alt-screen
  восстанавливает терминал на выходе.

## Headless e2e — тест интерактива через tmux
Интерактивный TUI нельзя протестировать без терминала — но tmux-пейн ЕСТЬ настоящий pty.
`scripts/tui-smoke.sh` (`bun smoke`) запускает реальный TUI в одноразовой tmux-сессии,
шлёт клавиши (`send-keys`) и снимает кадры (`capture-pane`) после каждого нажатия:

```
tmux new-session -d -s X -x 150 -y 34 'bun run src/cli.ts'
tmux send-keys -t X Down      # навигация
tmux send-keys -t X f         # fullscreen
tmux capture-pane -t X -p     # прочитать что нарисовалось
```
Безопасные клавиши только (nav / f / PgDn / q) — `s`/`r`/`D` не шлём, чтобы не трогать живые
сессии. Этот флоу ловит визуальные баги (ширина колонок, перенос строк, мангелинг контекста)
без живого терминала — то, что иначе ловит пользователь.

### Что смоук НЕ ловит — нужен глаз пользователя
`capture-pane -p` отдаёт **текстовую сетку символов**, а не картинку. Агент (Claude) видит
СТРУКТУРУ (что в какой клетке, перенос, выравнивание в предполагаемой моноширине), но **НЕ**
оценивает следующее — это судит только пользователь на своём терминале:
- **Цвет** — `-p` срезает ANSI-цвета. cyan/серый, контраст, не сливается ли — не видно.
- **Шрифт и ширина глифов** — box-drawing (`╭ │ ┃`), брайль-спиннер, вес `┃` vs `│`, эмодзи,
  широкие символы выравниваются по ШРИФТУ пользователя; в капче может быть ровно, у юзера —
  разъехаться. *(Реально кусало: ползунок `▐` выглядел «надкусанным» — поймал только юзер.)*
- **Размер терминала** — агент берёт свой (напр. 150×30); у юзера другой → иной перенос/колонки.
- **Анимация / мерцание / плавность** — капча статична; дёрганье и мигание не видны.
- **Мышь** — колесо/драг/клик в смоуке не пошлёшь; эти пути проверяются ЛОГИКОЙ, не глазами.

Вывод: смоук — для структурных багов. Эстетику (цвет, шрифт-зависимый рендер, ощущение анимации,
мышь) агент сужает ВСЛЕПУЮ по сетке+коду — финальное «красиво/ровно» подтверждает пользователь.
Не заявлять «выглядит красиво» как факт — писать «структурно ок, визуал на тебе».

## Производительность рендера и время жизни (горячий цикл / сироты)
> Реальный инцидент 2026-06-11: `ccmux-dev -f`, оставленный открытым, сжёг **~84% ядра 14 часов**.
> Два корня, оба теперь закрыты — но это легко вернуть регрессией, поэтому держим инварианты.

**Инвариант 1 — Ink-дерево перерисовывается только когда нужно.** В Ink ЛЮБОЙ `setState`
перерисовывает всё дерево (Yoga-layout + ANSI-diff), а fullscreen-транскрипт дорог (markdown-парс
на каждую видимую строку, расчёт высот). Поэтому:
- **Спиннер тикает ТОЛЬКО при активной работе.** `useSpinner(active)` запускает интервал лишь когда
  `items.some(it => it.status.active)` (есть «working»-сессия), и на 200мс, не 120мс. Idle-флот =
  статичный кадр, **ноль ререндеров**. *(Был безусловный `setInterval(…,120)` → 8.3 ререндера/сек
  вечно, даже когда всё простаивает — это и жгло ядро.)*
- **`ChatMessage` и `Markdown` мемоизированы** (`React.memo`). `ChatMessage` сравнивается по
  ИДЕНТИЧНОСТИ сообщения (`a.msg === b.msg`); спиннер-тик и fleet-поллинг (1.5с) перерисовывают App,
  но НЕ пересобирают массив сообщений → история не перерендерится и markdown не перепарсится. `spin`
  сравнивается только для `tool_call`-строк (анимированный глиф). `Markdown` мемоизирован как
  чистый parse-leaf.
- **`useTranscript` гейтит ре-парс по mtime** — перечитывает jsonl только когда файл изменился
  (idle-сессия = ноль работы вместо ре-парса 300 сообщений каждые 1.5с).

**Инвариант 2 — TUI умирает вместе с терминалом, не сиротеет.** Закрытие вкладки VS Code/iTerm
переусыновляет процесс к PID 1; без явного выхода render-loop (живой за счёт интервалов) рендерит в
мёртвый pty вечно. На дефолт сигналов полагаться НЕЛЬЗЯ — Ink/Bun их глушат (сирота игнорировал даже
`kill`/SIGTERM). `installExitOnTerminalDeath()` (`src/tui/run.tsx`) форсит `process.exit(0)` на
`SIGHUP`/`SIGTERM`, на первой же ошибке записи в stdout (EIO на мёртвом pty) и на EOF stdin.

### Как дебажить «комп горячий» / подозрение на горячий TUI
1. **Найти жор:** `ps aux | grep -E "src/cli.ts|ccmux.js" | grep -v grep` — смотри `%CPU` и колонку
   времени. `bun run src/cli.ts` на ~целом ядре часами = горячий цикл рендера.
2. **Сирота?** `ps -o pid,ppid,tty,stat -p <PID>` — `PPID=1` + `STAT` с `+` на мёртвом tty = процесс
   осиротел (терминал закрыт, а он жив). Это всегда баг — терминал должен был его убить.
3. **Убить:** обычный `kill` сирота может проглотить (сигналы перехвачены) → `kill -9 <PID>`. Это
   безопасно: сессиями рулит ДЕМОН (отдельный процесс, не Ink) — TUI лишь интерактивное меню.
4. **Замер инварианта:** запусти TUI в одноразовом tmux-пейне (`tmux new-session -d -s t 'bun run
   src/cli.ts -f'`), сэмпли `ps -o %cpu= -p <PID>` — на простаивающем флоте должно быть ~0%. Не ноль
   → проверь, не тикает ли спиннер без активных сессий (Инвариант 1).
5. **Проверка exit-on-death:** подними TUI в tmux-пейне, `tmux kill-session -t t`, через секунду
   `ps -p <PID>` — процесс ОБЯЗАН исчезнуть; в `~/.ccmux/ccmux.log` будет `terminal died — exiting`.

**Демон НЕ затронут** ничем из этого — он не крутит Ink. Горит/сиротеет только интерактивное меню.

## Модель IO/поллинга (mtime-кэш + видимый захват)
Если рендер-цикл холодный (раздел выше), второй источник нагрузки — **IO-слой поллинга**: каждые
1.5с (`useFleet`) и 4с (`useDiscover`) перечитывались и перепарсивались транскрипты, даже если
файлы не менялись, плюс форк `tmux capture-pane` на КАЖДУЮ запущенную сессию. Инварианты, чтобы
idle-флот делал почти ничего:

- **Один источник кэша — `src/util/mtimeCache.ts` (`MtimeCache`).** Кэширует по `(path, mtime)` и
  возвращает **ту же ссылку**, пока файл не менялся. Это даёт ДВА эффекта сразу: (1) нет повторного
  чтения/парса; (2) downstream `React.memo` (`SessionCard`, `ChatMessage`) видит стабильный
  `lastMessage` и бейлит. Ломается одно — ломается и memo. Консьюмеры: `lastTranscriptMessage`,
  `sessionUsedTokens` (`agent/index.ts`), `discoverActive` (`tui/discover.ts`).
- **Никаких полных чтений транскрипта.** Большие jsonl читаются хвостом (`readTailLines`, чанки с
  конца) + при нужде головой (`readHead`, 64КБ — только для cwd). `discover` больше НЕ делает
  `readFileSync` целиком (было доминирующей IO-стоимостью на мегабайтных файлах).
- **Захват пейна — только для видимых карточек.** `collectRows(m, { liveNames })`: TUI передаёт имена
  видимых (окно скролла) + выбранной сессии; их пейны форкаются (`capture-pane`), остальные запущенные
  переиспользуют последний скан из `scanCache` (`commands/list.ts`). Оффскрин-состояние невидимо и
  освежается при скролле карточки в окно. **CLI `list` зовёт `collectRows` БЕЗ `liveNames` → захват
  всех** (корректность таблицы). Форки масштабируются по экрану, не по размеру флота (важно для 20+).
- **Высоты сообщений кэшируются** (`chatMessageHeight`, WeakMap по `msg`×`width`) — `sanitize`+`wrap`
  не гоняются в measure-циклах транскрипта на каждый ререндер.

### Дебаг IO-нагрузки
- Idle-флот должен давать **ноль** дисковых чтений транскриптов между записями — если `discover`/`list`
  читают мегабайты, проверь, что путь идёт через `MtimeCache` (не зови `readFileSync` на jsonl напрямую).
- Карточки не должны перерисовываться на поллинг, если контент тот же — сломанный memo обычно значит,
  что `lastMessage` потерял стабильную ссылку (консьюмер обошёл `MtimeCache`) или компаратор
  `SessionCard` не покрыл новое рендер-поле.
- Лишние `tmux`-форки — проверь, что `liveNames` доходит до `collectRows` (через `liveNamesRef` в
  `App.tsx` → `useFleet`); пустой/отсутствующий → захват всех.
